# Lint 工作流：知识库健康检查

> 本文档供 LLM 执行 `/lint` 命令时参考。完整执行以下 7 项检查，输出结构化报告。

---

## 触发方式

```
/lint              ← 手动触发完整健康检查
/lint --quick      ← 快速检查（仅前 3 项）
```

也会在每 5 次 `/ingest` 后自动提醒用户运行。

---

## 完整流程

### 准备：读取知识库全貌

```bash
# 读取内容目录
cat "{{WIKI_PATH}}/wiki/index.md"

# 获取所有 wiki 页面列表
qmd ls

# 读取最近操作日志
grep "^## \[" "{{WIKI_PATH}}/wiki/log.md" | tail -20
```

---

### 检查项 1：矛盾论断检测

**目标**：找出不同页面之间相互冲突的论断。

**执行方法**：

1. 读取所有 `concepts/` 和 `synthesis/` 页面
2. 对比同一主题在不同资料中的论断
3. 查找已有的 `⚠️ 矛盾` 标注，确认是否已解决或需更新

**常见矛盾模式**：

```bash
# 搜索已标注的矛盾
qmd search "⚠️ 矛盾" --collection {{WIKI_NAME}}
```

**矛盾严重性分级**：

| 级别 | 描述 | 处理建议 |
|------|------|---------|
| 🔴 高 | 核心结论互相否定 | 立即标注，建议用户澄清 |
| 🟡 中 | 侧重点不同，未直接冲突 | 标注差异，保留两种视角 |
| 🟢 低 | 可能因情境/时间不同 | 添加上下文注释 |

**输出格式**：

```
### 矛盾论断（N 处）

🔴 **高优先级矛盾**：
- **支付安全性态度**
  - [[sources/用户访谈-2024-03]] 中用户 B 表示"快捷支付方便"，未提安全顾虑
  - [[sources/竞品分析-2024-02]] 指出竞品快捷支付因安全顾虑接受度低
  - 📍 位置：[[concepts/支付流程痛点]]、[[sources/竞品分析-2024-02]]
  - 🔧 建议：在下次访谈中直接询问用户对安全性的看法
```

---

### 检查项 2：过时论断检测

**目标**：找出被更新资料取代但未更新的旧论断。

**执行方法**：

1. 检查 `concepts/` 和 `synthesis/` 中带有早期日期的论断
2. 对比同一概念的不同来源日期
3. 识别已被新资料明确否定的旧结论

**过时标志**：

```bash
# 查找可能过时的内容标记
qmd search "🔄 已更新" --collection {{WIKI_NAME}}
# 查找较旧的结论（通过 date 字段）
qmd search "date: 2023" --collection {{WIKI_NAME}}
```

**输出格式**：

```
### 过时论断（N 处）

🔄 **可能过时**：
- **导航设计偏好**
  - [[concepts/导航痛点]] 中 2023-06 的结论"用户偏好底部导航"
  - 但 [[sources/用户访谈-2024-01]] 中用户表示"顶部导航也没问题"
  - 📍 需要更新：[[concepts/导航痛点]] 第 3 节
  - 🔧 建议：将旧结论标注为"历史观点"，添加最新来源
```

---

### 检查项 3：孤立页面检测

**目标**：找出没有任何入站链接（被其他页面引用）的页面。

**执行方法**：

```bash
# 列出所有 wiki 页面
qmd ls --collection {{WIKI_NAME}}

# 对每个页面，搜索其他页面中是否有引用
# 例如检查 "用户-C" 是否被引用：
qmd search "用户-C" --collection {{WIKI_NAME}}
```

**孤立页面处理建议**：

| 类型 | 建议 |
|------|------|
| 新建的 entities 页面 | 检查相关 concepts 是否应添加引用 |
| 旧的 sources 摘要 | 确认是否已在 index.md 中 |
| 早期的 synthesis 页面 | 检查是否需要与新内容关联 |

**输出格式**：

```
### 孤立页面（N 个）

📄 **以下页面没有被其他页面引用**：
- `wiki/entities/用户-D.md`（创建于 2024-01-15，仅在 sources/ 摘要中提及）
  - 🔧 建议：在 [[concepts/目标用户群]] 中添加引用
- `wiki/sources/竞品分析-2023-11.md`（最旧资料，可能已无当前引用）
  - 🔧 建议：在 [[synthesis/竞品全景]] 中添加历史参考引用
```

---

### 检查项 4：缺失页面检测

**目标**：找出在其他页面中被 `[[引用]]` 但实际上不存在的页面。

**执行方法**：

```bash
# 提取所有 [[引用]] 链接
grep -r "\[\[" "{{WIKI_PATH}}/wiki/" | grep -oP '\[\[\K[^\]]+' | sort | uniq
# 对比实际存在的文件列表
qmd ls --collection {{WIKI_NAME}}
```

**输出格式**：

```
### 缺失页面（N 个）

🔗 **被引用但不存在的页面**：
- `[[concepts/用户决策模型]]`（在 [[synthesis/支付优化建议]] 和 [[concepts/支付流程痛点]] 中被引用）
  - 🔧 建议：导入相关资料后创建，或标注为"计划中"
- `[[entities/产品-竞品C]]`（在 [[sources/竞品分析-2024-02]] 中被引用）
  - 🔧 建议：创建竞品 C 的实体页面
```

---

### 检查项 5：缺失交叉引用检测

**目标**：找出语义高度相关但没有互相链接的页面对。

**执行方法**：

```bash
# 使用语义搜索找相似页面
qmd vsearch "支付流程" --collection {{WIKI_NAME}} --top 5
# 检查这些页面之间是否有交叉引用
```

**典型缺失引用场景**：

- `concepts/加载速度` 与 `concepts/支付流程痛点` 高度相关但没有互链
- `entities/用户-A` 出现在多个 sources 摘要中，但 sources 没有引用 entities
- `synthesis/` 页面没有引用支撑它的 `sources/` 页面

**输出格式**：

```
### 缺失交叉引用（N 对）

🔗 **应该互相链接但没有链接的页面**：
- `[[concepts/加载速度]]` ↔ `[[concepts/支付流程痛点]]`
  - 理由：支付痛点的主要原因之一是加载速度，两者高度相关
  - 🔧 建议：在两个页面的 related 部分和正文中添加互链
- `[[entities/用户-A]]` ← `[[sources/用户访谈-2024-01]]`
  - 理由：用户 A 出现在两份访谈摘要中，但 2024-01 摘要未链接到用户 A 的实体页
  - 🔧 建议：在 [[sources/用户访谈-2024-01]] 中添加对 [[entities/用户-A]] 的引用
```

---

### 检查项 6：数据空白检测

**目标**：识别知识库中缺失的重要信息，为未来研究方向提供建议。

**执行方法**：

1. 扫描 `concepts/` 页面中的 `❓ 待验证` 标注
2. 识别只有 1 个来源支撑的重要结论（单一来源风险）
3. 识别在某个维度完全没有数据的实体或概念

**常见数据空白类型**：

| 类型 | 描述 | 优先级 |
|------|------|--------|
| 未验证假设 | 只被 1 份资料提及的重要结论 | 高 |
| 用户群缺失 | 某类用户（如老年用户）没有任何访谈记录 | 中 |
| 竞品盲区 | 关键竞品没有分析资料 | 中 |
| 时间断层 | 某主题最后一份资料超过 6 个月 | 低 |

#### 数据空白的类型

| 类型 | 描述 | 优先级 |
|------|------|--------|
| 缺失实体 | 重要但未被研究的用户群体、竞品或产品功能 | 高 |
| 缺失维度 | 已有实体但缺少关键维度的数据 | 中 |
| 网络可补充 | 可以通过搜索引擎或学术搜索填补的公开信息（如行业基准数据、公开报告） | 中 |
| 时效性缺失 | 现有数据已过时，需要最新资料 | 高 |

LLM 会主动建议可以通过网络搜索（如 Google Scholar、行业报告网站）补充的信息方向，帮助用户发现需要寻找的新资料来源。

**输出格式**：

```
### 数据空白（N 处）

❓ **需要更多数据支撑的论断**：
- **快捷支付安全性态度**
  - 当前仅有 1 份资料间接提及
  - 🔧 建议：在下次用户访谈中专门询问对支付安全性的看法

📭 **未覆盖的重要领域**：
- **40岁以上用户群**：所有访谈对象均为 28-35 岁，无老年用户数据
  - 🔧 建议：寻找或开展针对此年龄段的研究
- **Android 用户体验**：所有访谈均为 iOS 用户
  - 🔧 建议：招募 Android 用户进行访谈或问卷调查
```

---

### 检查项 7：log.md 完整性检查

**目标**：确认日志格式正确，操作记录完整。

```bash
# 验证日志格式
grep "^## \[" "{{WIKI_PATH}}/wiki/log.md"

# 统计各类操作数量
grep "^## \[" "{{WIKI_PATH}}/wiki/log.md" | grep "ingest" | wc -l
grep "^## \[" "{{WIKI_PATH}}/wiki/log.md" | grep "query" | wc -l
grep "^## \[" "{{WIKI_PATH}}/wiki/log.md" | grep "lint" | wc -l
```

**检查内容**：
- 所有日志条目是否符合格式 `## [YYYY-MM-DD] type | 标题`
- index.md 的页面统计数字是否与实际一致
- 是否有 ingest 操作没有对应的日志记录

## 在 log.md 追加记录

Lint 完成后，在 `log.md` 末尾追加：

```markdown
## [YYYY-MM-DD] lint | N issues found

- 执行检查：7 项
- 发现问题：N 处（矛盾: x | 孤立页面: x | 缺失引用: x | 数据空白: x）
- 知识库页面数：N（sources: x | entities: x | concepts: x | synthesis: x）
```

**格式说明**：
- `N issues found` 是问题总数，方便后续用 grep 统计知识库健康趋势
- 支持 `grep "^## \[" log.md | grep "lint"` 解析所有 lint 历史
