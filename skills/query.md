# Query 工作流：查询知识库

> 本文档供 LLM 执行 `/query` 命令时参考。每次查询时按以下流程执行。

---

## 触发方式

```
/query 我们的用户在支付流程中最大的痛点是什么？
/query 竞品在导航设计上做了哪些不同的选择？
/query 新功能 X 的用户接受度如何？满意度如何？
```

---

> ⚠️ **重要约束**：查询时，**必须先通过 qmd 搜索来发现相关文件，不得凭借对 index.md 的已知内容直接选择文件**。qmd 能发现关键词不明显但语义相关的页面（如 `social-play-motivation` 对"山头服用户为什么流失"语义相关但无直接关键词），而手工选择文件存在遗漏风险。

## 完整流程

### 步骤 1：通过 qmd 搜索发现相关文件（必须执行，不可跳过）

> ❌ **禁止**：凭借对 index.md、log.md 或之前对话的已知内容，直接选择文件读取。
> ✅ **必须**：将用户查询交给 qmd，由搜索引擎客观排序后，再决定读哪些文件。

**首先确认向量索引状态**：

```bash
node <SKILL_PATH>/scripts/qmd/dist/cli/qmd.js status
```

查看 `status` 输出中 `Vectors` 一行：
- `Vectors: N embedded`（N > 0）→ 使用混合搜索 `qmd query`
- `Vectors: 0 embedded` → 使用 BM25 搜索 `qmd search`（向量索引未就绪）

**有向量索引时——执行混合搜索**（BM25 + 向量语义 + LLM 重排序）：

```bash
node <SKILL_PATH>/scripts/qmd/dist/cli/qmd.js query "{{用户的查询关键词}}" -c {{WIKI_NAME}} -n 10
```

> ⚠️ **注意**：`qmd query` 在向量索引未就绪时会**静默返回空**（无任何输出或报错），这是 qmd 当前版本的特性。如果运行后无输出，请先检查 `qmd status` 确认向量索引状态。

**无向量索引时——使用 BM25 关键词搜索**：

```bash
node <SKILL_PATH>/scripts/qmd/dist/cli/qmd.js search "{{关键词1}} {{关键词2}}" -c {{WIKI_NAME}} -n 10
```

若 BM25 搜索也无结果，尝试调整关键词（中英文均试）或扩宽搜索词范围。

**如果向量索引未就绪**，告知用户：
```
⚠️ 向量索引未就绪，本次使用 BM25 关键词搜索（仍可找到相关内容，但语义匹配能力较弱）。
如需完整语义搜索，请运行：
  node <SKILL_PATH>/scripts/qmd/dist/cli/qmd.js embed
```

**搜索结果即为候选文件列表**，按相关性得分排序。取 top 5~10 进入下一步。

---

### 步骤 2：读取 index.md（辅助，仅用于补全盲区）

读取搜索结果后，**可选**地扫描 `index.md`，检查是否有 qmd 搜索遗漏的明显相关页面：

```bash
cat "{{WIKI_PATH}}/wiki/index.md"
```

> 注意：index.md 的作用是**补充 qmd 可能的盲区**，而不是替代 qmd 的搜索结果。
> 如果 qmd 已返回 10 个相关结果，可跳过此步骤。

---

### 步骤 3：读取 qmd 返回的候选文件

根据步骤 1 的搜索结果，读取最相关的页面：

```bash
# 读取单个页面
qmd get wiki/concepts/支付流程痛点.md

# 批量读取
qmd multi-get "wiki/concepts/支付流程痛点.md,wiki/entities/用户-A.md,wiki/synthesis/支付优化建议.md"

# 读取页面特定段落（行号）
qmd get wiki/concepts/支付流程痛点.md:50 -l 100
```

---

### 步骤 5：生成综合答案

基于读取的页面内容，生成高质量的综合答案。

**答案的核心要求**：
1. **带引用**：每个论断都要注明来自哪个页面（哪份原始资料）
2. **综合多来源**：不偏向单一来源，综合所有相关证据
3. **标注矛盾**：如果不同来源有冲突，明确指出
4. **说明置信度**：区分"多来源强支撑"和"单一来源初步发现"

**答案格式示例（Markdown 页面）**：

```markdown
## 支付流程用户痛点分析

基于 3 份资料（2 次用户访谈 + 1 份竞品分析），支付流程痛点主要集中在两个维度：

### 1. 步骤数量过多（置信度：高，2/2 访谈提及）

用户普遍反映支付流程需要"太多步骤"：
- 用户 A 明确表示需要 3 步确认才能完成支付（[[sources/用户访谈-2024-03]]）
- 用户 C 同样提到"流程太繁琐"（[[sources/用户访谈-2024-01]]）

> 💡 与竞品对比：主要竞品已将支付步骤压缩至 2 步（[[sources/竞品分析-2024-02]]）

### 2. 加载速度慢（置信度：高，2/2 访谈提及）

加载性能问题在两次访谈中均被独立提及：
- "有时候点了没反应" —— 用户 A（[[sources/用户访谈-2024-03]]）
- 用户 C 描述为"页面卡顿"（[[sources/用户访谈-2024-01]]）

> ⚠️ **矛盾**：竞品分析显示，快捷支付功能在竞品中因安全顾虑接受度低
> （[[sources/竞品分析-2024-02]]），但我们的用户对此未表达明显担忧。
> 安全性认知差异的原因尚待研究。

### 总结

支付优化需要"步骤简化"与"性能提升"双管齐下，两者相互影响用户感知。
建议参考：[[synthesis/支付优化建议]]
```

---

### 步骤 6：根据用户需求调整答案形式

根据用户的实际需求，可以将答案以不同形式呈现：

#### 形式 A：Markdown 摘要页面（默认）
见上方示例。适合深度分析和需要归档的场景。

#### 形式 B：对比表格

```markdown
| 维度 | 我们的产品 | 竞品 A | 竞品 B |
|------|-----------|--------|--------|
| 支付步骤数 | 3 步 | 2 步 | 1 步 |
| 加载时间 | 3-5s | <2s | <1s |
| 用户满意度 | 中（访谈反馈） | 高（[[sources/竞品分析-2024-02]]） | 未知 |
```

#### 形式 C：Marp 幻灯片格式

```markdown
---
marp: true
theme: default
---

# 支付流程痛点分析

**来源**：2 次用户访谈 + 1 份竞品分析

---

## 核心发现

1. **步骤过多**：平均 3 步，竞品已做到 2 步
2. **加载慢**：2/2 用户自发提及
3. **机会**：快捷支付方向正确，需解决性能问题

---
```

#### 形式 D：行动项清单

```markdown
## 基于研究洞察的行动建议

### 高优先级
- [ ] 减少支付步骤至 2 步（有明确用户诉求）
- [ ] 优化支付页面加载性能（目标 <2s）

### 中优先级
- [ ] 研究用户对安全性的实际担忧（数据空白）
```

#### 形式 E：数据可视化（图表）

如果分析涉及定量数据或趋势对比，可以生成 Python 图表代码（matplotlib/seaborn）：

```python
import matplotlib.pyplot as plt
import matplotlib
matplotlib.rcParams['font.sans-serif'] = ['SimHei']  # 中文支持

labels = ['操作步骤', '加载速度', '信任感', '信息清晰度']
scores = [3.2, 2.8, 4.1, 3.5]
plt.bar(labels, scores, color=['#e74c3c' if s < 3.5 else '#2ecc71' for s in scores])
plt.title('支付流程各维度用户满意度')
plt.ylabel('满意度（1-5）')
plt.savefig('wiki/synthesis/assets/payment-satisfaction.png', dpi=150)
plt.show()
```

#### 形式 F：Obsidian Canvas（画布）

对于需要可视化关系网络的分析，可以生成 `.canvas` 文件（Obsidian Canvas JSON 格式），在 Obsidian 中以画布方式展示概念之间的关系图。适合梳理用户旅程、竞品关系、因果链路等复杂关联。

完成回答后，询问用户：

```
这次分析是否值得保存到知识库？

如果是，我将把这份分析归档到 wiki/synthesis/，
方便下次查询时直接引用，避免重复分析工作。

归档到：wiki/synthesis/支付流程痛点分析.md
确认归档？（是/否，或输入不同的文件名）
```

### 步骤 9：在 log.md 追加记录（仅归档时）

如果执行了归档，在 `log.md` 末尾追加：

```markdown
## [YYYY-MM-DD] query | 支付流程用户痛点

- 查询问题：我们的用户在支付流程中最大的痛点是什么？
- 搜索方式：qmd query（混合搜索）
- 涉及页面：sources/用户访谈-2024-03、sources/用户访谈-2024-01、sources/竞品分析-2024-02

```

**注**：如果用户选择不归档，不记录到 log.md（避免日志膨胀）。

> 💡 **探索的复利效应**：通过归档机制，用户的每一次探索性提问都会像导入新资料一样，在知识库中不断积累复利——对比分析、发现的关联、分析结论，这些有价值的洞察不会消失在聊天记录中，而是成为知识库永久的一部分。

## qmd 搜索最佳实践

### 搜索词建议

| 查询目标 | 推荐搜索词 |
|---------|-----------|
| 特定功能的用户反馈 | `"功能名 用户反馈"` 或 `"功能名 体验"` |
| 竞品对比信息 | `"竞品名"` 或 `qmd search "competitor"` |
| 某用户的完整画像 | `qmd get entities/用户-A.md` |
| 某概念的所有相关内容 | `qmd query "概念名"` 再 `qmd multi-get` |

### 搜索无结果时

如果 qmd 未返回相关结果：

1. 尝试换用不同关键词（中英文均试）
2. 降低搜索范围，搜索更宽泛的概念
3. 直接读取 `wiki/index.md`，手动定位相关页面
4. 告知用户："目前知识库中尚无关于 X 的记录，建议导入相关资料后再查询"

### HYDE 搜索（高级）

对于复杂的假设性问题，可以使用 Hyde 搜索：

```bash
qmd query "如果要改进支付流程，最关键的因素是什么" --type hyde
```

Hyde 搜索会先生成假设性答案，再用答案作为查询向量，适合探索性研究问题。
