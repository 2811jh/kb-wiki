# Query 工作流：查询知识库

> 本文档供 LLM 执行 `/query` 命令时参考。每次查询时按以下流程执行。

---

## 触发方式

```
/query 我们的用户在支付流程中最大的痛点是什么？
/query 竞品在导航设计上做了哪些不同的选择？
/query 新功能 X 的用户接受度如何？
```

---

## 完整流程

### 步骤 1：分析查询意图，选择搜索策略

根据查询类型选择最合适的 qmd 搜索命令：

| 查询类型 | 推荐命令 | 适用场景 |
|---------|---------|---------|
| 精确关键词 | `qmd search <关键词>` | 搜索特定术语、人名、产品名 |
| 语义搜索 | `qmd vsearch <描述>` | 搜索概念、问题、模糊描述 |
| 综合最佳 | `qmd query <问题>` | 大多数查询的推荐选择 |

**对于大多数研究问题，优先使用 `qmd query`**（混合搜索 + 重排序 + 上下文扩展）。

---

### 步骤 2：读取 index.md 定位相关页面

在执行搜索前，先读取 `wiki/index.md` 了解知识库全貌：

```bash
cat "{{WIKI_PATH}}/wiki/index.md"
```

根据 index.md 的分类，判断哪些页面可能与查询相关，缩小搜索范围。

---

### 步骤 3：执行 qmd 搜索

**基础查询（推荐）**：

```bash
qmd query "支付流程用户痛点" --collection ux-research
```

**精确关键词搜索**：

```bash
qmd search "支付流程" --collection ux-research
```

**语义搜索**：

```bash
qmd vsearch "用户在付款时遇到的阻碍和挫折" --collection ux-research
```

**跨集合搜索**（有多个知识库时）：

```bash
qmd query "支付流程" --collections ux-research,product-wiki
```

**带重排序**（提升结果质量）：

```bash
qmd query "支付流程痛点" --rerank
```

---

### 步骤 4：读取相关页面内容

根据搜索结果，读取最相关的页面：

```bash
# 读取单个页面
qmd get wiki/concepts/支付流程痛点.md

# 批量读取
qmd multi-get "wiki/concepts/支付流程痛点.md,wiki/entities/用户-A.md,wiki/synthesis/支付优化建议.md"

# 读取页面特定段落（行号）
qmd get wiki/concepts/支付流程痛点.md:50 -l 100
```

---

### 步骤 5：生成综合答案

基于读取的页面内容，生成高质量的综合答案。

**答案的核心要求**：
1. **带引用**：每个论断都要注明来自哪个页面（哪份原始资料）
2. **综合多来源**：不偏向单一来源，综合所有相关证据
3. **标注矛盾**：如果不同来源有冲突，明确指出
4. **说明置信度**：区分"多来源强支撑"和"单一来源初步发现"

**答案格式示例（Markdown 页面）**：

```markdown
## 支付流程用户痛点分析

基于 3 份资料（2 次用户访谈 + 1 份竞品分析），支付流程痛点主要集中在两个维度：

### 1. 步骤数量过多（置信度：高，2/2 访谈提及）

用户普遍反映支付流程需要"太多步骤"：
- 用户 A 明确表示需要 3 步确认才能完成支付（[[sources/用户访谈-2024-03]]）
- 用户 C 同样提到"流程太繁琐"（[[sources/用户访谈-2024-01]]）

> 💡 与竞品对比：主要竞品已将支付步骤压缩至 2 步（[[sources/竞品分析-2024-02]]）

### 2. 加载速度慢（置信度：高，2/2 访谈提及）

加载性能问题在两次访谈中均被独立提及：
- "有时候点了没反应" —— 用户 A（[[sources/用户访谈-2024-03]]）
- 用户 C 描述为"页面卡顿"（[[sources/用户访谈-2024-01]]）

> ⚠️ **矛盾**：竞品分析显示，快捷支付功能在竞品中因安全顾虑接受度低
> （[[sources/竞品分析-2024-02]]），但我们的用户对此未表达明显担忧。
> 安全性认知差异的原因尚待研究。

### 总结

支付优化需要"步骤简化"与"性能提升"双管齐下，两者相互影响用户感知。
建议参考：[[synthesis/支付优化建议]]
```

---

### 步骤 6：根据用户需求调整答案形式

根据用户的实际需求，可以将答案以不同形式呈现：

#### 形式 A：Markdown 摘要页面（默认）
见上方示例。适合深度分析和需要归档的场景。

#### 形式 B：对比表格

```markdown
| 维度 | 我们的产品 | 竞品 A | 竞品 B |
|------|-----------|--------|--------|
| 支付步骤数 | 3 步 | 2 步 | 1 步 |
| 加载时间 | 3-5s | <2s | <1s |
| 用户满意度 | 中（访谈反馈） | 高（[[sources/竞品分析-2024-02]]） | 未知 |
```

#### 形式 C：Marp 幻灯片格式

```markdown
---
marp: true
theme: default
---

# 支付流程痛点分析

**来源**：2 次用户访谈 + 1 份竞品分析

---

## 核心发现

1. **步骤过多**：平均 3 步，竞品已做到 2 步
2. **加载慢**：2/2 用户自发提及
3. **机会**：快捷支付方向正确，需解决性能问题

---
```

#### 形式 D：行动项清单

```markdown
## 基于研究洞察的行动建议

### 高优先级
- [ ] 减少支付步骤至 2 步（有明确用户诉求）
- [ ] 优化支付页面加载性能（目标 <2s）

### 中优先级
- [ ] 研究用户对安全性的实际担忧（数据空白）
```

---

### 步骤 7：询问用户是否将洞察归档

完成回答后，询问用户：

```
这次分析是否值得保存到知识库？

如果是，我将把这份分析归档到 wiki/synthesis/，
方便下次查询时直接引用，避免重复分析工作。

归档到：wiki/synthesis/支付流程痛点分析.md
确认归档？（是/否，或输入不同的文件名）
```

---

### 步骤 8：归档洞察（如用户确认）

如果用户选择归档，在 `wiki/synthesis/` 创建新页面：

```markdown

date: 2024-03-20

---

# 支付流程痛点分析

> **查询**：我们的用户在支付流程中最大的痛点是什么？
> **生成日期**：2024-03-20
> **来源页面**：3 个（sources: 3）

[完整分析内容...]
```

同时更新 `wiki/index.md`，在 synthesis/ 部分添加条目：

```

```

如果执行了归档，在 `log.md` 末尾追加：

```markdown

```

## qmd 搜索最佳实践

### 搜索词建议

| 查询目标 | 推荐搜索词 |
|---------|-----------|
| 特定功能的用户反馈 | `"功能名 用户反馈"` 或 `"功能名 体验"` |
| 竞品对比信息 | `"竞品名"` 或 `qmd search "competitor"` |
| 某用户的完整画像 | `qmd get entities/用户-A.md` |
| 某概念的所有相关内容 | `qmd query "概念名"` 再 `qmd multi-get` |

### 搜索无结果时

如果 qmd 未返回相关结果：

1. 尝试换用不同关键词（中英文均试）
2. 降低搜索范围，搜索更宽泛的概念
3. 直接读取 `wiki/index.md`，手动定位相关页面
4. 告知用户："目前知识库中尚无关于 X 的记录，建议导入相关资料后再查询"

### HYDE 搜索（高级）

对于复杂的假设性问题，可以使用 Hyde 搜索：

```bash
qmd query "如果要改进支付流程，最关键的因素是什么" --type hyde
```

Hyde 搜索会先生成假设性答案，再用答案作为查询向量，适合探索性研究问题。
